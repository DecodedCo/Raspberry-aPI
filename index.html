Decoded API Documentation

The Brief

Our brief requires us to store some data about checkins. We can't just store this in our user's browser for two reasons:

  1. Data stored in the browser is temporary; if the user closes their browser or deletes their cookies, the data is lost.
  2. We want a common data store; a central repository of all user checkins. We will then be able to access all of this data if we want to keep track of checkins.

So, we need to store our data on a single computer that we can control — a web server.

Our server

A server might be provided by someone like Amazon Web Services, but we chose a Raspberry Pi — a credit card-sized computer that you can buy for $25. It's sitting in the corner of our London HQ!

We chose a Raspberry Pi to demonstrate that there's nothing magical about a web-server. In the real world, you want to use something a bit more sturdy. If someone spills tea on the Pi, or unplugs it, or our internet connection breaks â€“ the server will stop working!

Don't worry, we have a back-up at: form.decoded.co:3000/checkin

There are lots! In fact, you could use any programming language on the server. You are in charge of the environment, so it's up to you to make it work. This is in contrast with front-end technologies: browsers are designed to be universal tools, so there are standards about what languages are able to be processed by the browser (we typically only use three languages in the browser: HTML, CSS and JavaScript). We chose Node.js, because it allows us to create a server very easily. Plus it uses JavaScript, the final language you will be using today. So if you do want to look into our server-side code later, you can!

With a few lines of code, your app can send data to this server to be stored. It can also pull data down from the server to be displayed in your app.

What data is stored in the API?

Your app asks users to input their Twitter username and press return, it then displays their username and the number of times they have checked in. For this to work, we need to store a list of usernames with a corresponding number of checkins.

How is the data stored in the API?

The data is stored in a JSON format, take a look at the example below.

{
	decodedco: 1, 
	johnridpath: 2,
	kathrynparsons: 3,
	aliblackwell: 4
}

Where is the data stored in the API?

Each workshop will have a unique name assigned to it, usually the date of the session (e.g. 01-04-13). That data will be accessible at the following URL: http://api.decoded.co/checkin/01-04-13.

The first time someone checks into one of the apps created during your session a new datastore will automatically be created - this initial datastore will then be used to store the data for everyone's apps.

To make a checkin, you can append a query to this URL using the "username" parameter. For example: http://api.decoded.co/checkin/01-04-13?username=kathrynparsons.

Submitting data from your app

You have a few options as to how you might submit data to a datastore in this api.

1) Form

The simplest way is to use an HTML form with a single input. To choose where to send the data, you will need to set the form's "action" attribute to your session's API URL. Your form should include a single text input box with the "name" attribute set to "username". For example:

<form action="http://api.decoded.co/checkin/04-07-13">
	<input type="text" name="username">
</form>
2) AJAX

You can also submit data via AJAX. AJAX can send data to and retrieve data from the server without leaving the current page. We recommend using jQuery's AJAX function, as follows:

$.ajax({
	url: 'http://api.decoded.co/checkin/04-07-13',
	data: {username: "DecodedCo"},
	method: 'POST',
	dataType: 'jsonp',
	jsonp: 'callback',
	jsonpCallback: 'jsonpCallback',
	success: function(data){
		console.log(data);
	}
});

For more information, please read the jQuery .ajax() documentation (http://api.jquery.com/jQuery.ajax/).
